# 需求-项目重构最佳实践
fab-demo 项目的src目录的内容，需要重新梳理，并优化。
按照React + Vite + shadcn/ui 的技术栈中的最佳设计实践，目录结构应该是这样
```plaintext
src/
├── assets/              # 静态资源 (图片, svg)
├── components/          # 组件层
│   ├── ui/              # [shadcn] 基础原子组件 (Button, Input, Card) - 别动逻辑
│   ├── common/          # 全局通用的业务组件 (Header, Footer, UserAvatar)
│   └── providers/       # 全局 Context Providers (Theme, Auth, Toast)
├── features/            # [核心] 按功能模块划分 (比如版本管理模块)
│   └── VersionManager/  # 示例：版本管理
│       ├── components/  # 该模块专用组件 (VersionCard, TimelineNode)
│       ├── types.ts     # 模块专用类型定义
│       └── index.tsx    # 模块主入口
├── hooks/               # 全局通用 Hooks (use-mobile, use-debounce)
├── lib/                 # 工具库
│   └── utils.ts         # [shadcn] cn() 函数存放处
├── panels/              # 独立面板，或者页面， (只负责组装 feature 和 layout)
├── services/            # API 请求层 (axios/fetch 封装)
|—— data/                # mock数据
├── stores/              # 全局状态管理 (Zustand/Redux)
├── App.tsx              # 路由配置入口
└── main.tsx             # 入口文件
```

## 核心模块详细解析
1. 基础 UI 与 业务组件的分离 (components/)
这是使用 shadcn/ui 最需要注意的地方：

* components/ui/ (原子层)：
    * 原则：尽量不要修改这里的逻辑，只修改样式（通过 Tailwind）。它们应该是纯净的、无业务逻辑的“哑组件”。
    例子：Button, Input, Sheet, Dialog.

* components/common/ (分子层)：

    * 由多个原子组件组成的、全站通用的复合组件。
        例子：PageHeader (包含面包屑和标题), DataTable (封装了 Table 和 Pagination), ModeToggle (主题切换器).


2. Panels (panels/)= 一个完整的小功能模块 UI。通常是叫Pages,但是本项目为单页应用，这个模块为Panel,即项目导航目录。

3. 特性模块化 (features/) —— 进阶推荐。当项目变大时，按类型分目录（所有组件放一起，所有 Hook 放一起）会变得难以维护。按功能分目录（Co-location）是更好的选择。比如VersionManager功能比较多，但是又只属于某个页面的一部分，因此单独拆分为一个特性模块。

4. 项目级别状态管理 (stores/) —— 进阶推荐。当项目变大时，使用全局状态管理工具（Redux, Zustand, Jotai）来管理复杂的状态交互。

## 注意：
1. 善用 Alias (@)：在 vite.config.ts 和 tsconfig.json 配置路径别名，用 @/components/ui/button 引用，避免 ../../../../ 地狱。
2. 严格隔离 components/ui：把它当作第三方库对待，不要往里面写业务逻辑。
3. Feature First：如果你的面板功能很复杂（比如电商后台），请务必使用 features/ 目录结构。
4. 所有的改动只是迁移位置，千万不要破坏现有的页面呈现。
5. tsx组件文件命名，尽量用大驼峰
6. 自定义 Hooks ：小驼峰 (camelCase)
7. 工具函数与库 (Utils / Lib) ：小驼峰 (camelCase)
